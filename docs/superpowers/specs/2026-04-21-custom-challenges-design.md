# Custom Challenges — Design

**Date:** 2026-04-21
**Branch:** `td-custom-challenges`

## Goal

Let users add their own training challenges dynamically, generated by an LLM of their choice. The user copies a prologue (grammar + validation regexes) into their LLM, asks for a song/exercise, pastes the output back into the app, and the challenge appears in the training picker alongside built-ins.

## Scope

**In:** DSL, parser, add dialog, edit dialog, delete, localStorage persistence, "Custom" group in the training picker, tests.

**Out (deferred):** file/URL import, cross-device sync, sharing, renaming-without-re-paste, editing by splitting an existing block.

## DSL

A line-based format. `key: value`. A new `title:` line starts a new challenge (natural delimiter — no blank-line rule required, though blank lines are allowed between blocks for readability).

### Fields

| Key | Required | Meaning |
|---|---|---|
| `title` | yes | 1-60 printable chars; starts a new block |
| `group` | yes | `long-tones` \| `scales` \| `melodies` |
| `description` | no | up to 120 chars, 1 line |
| `transpose` | no | integer semitones; default `-2` (Bb trumpet written) |
| `notes` | yes | space-separated `<NOTE>` tokens; ≥1 required |

### Note grammar

```
<NOTE>  := <PITCH> | -
<PITCH> := [A-G] (b | #)? [0-8]    (scientific pitch; C4 = middle C)
<REST>  := -                        (single hyphen)
```

### Per-line validation regexes (also shown in the prologue for LLM self-check)

```
^title:\s+\S.{0,59}$
^group:\s+(long-tones|scales|melodies)$
^description:\s+\S.{0,119}$
^transpose:\s+-?\d+$
^notes:\s+([A-G][b#]?[0-8]|-)(\s+([A-G][b#]?[0-8]|-))*$
```

### Semantics

- A `<PITCH>` parses to **written** MIDI using scientific pitch notation: `(octave+1)*12 + step + accidental`, where step is `{C:0, D:2, E:4, F:5, G:7, A:9, B:11}` and accidental is `-1` for `b`, `+1` for `#`, `0` otherwise.
- **Stored concert MIDI = written MIDI + transpose.** Default `transpose: -2` matches a Bb trumpet part (written C sounds as concert Bb). Users who want literal concert pitch specify `transpose: 0`.
- `-` becomes `{kind: "rest"}`.
- **Range clamp:** concert MIDI must land in `[36, 96]` (low C2 to high C7 concert). Out-of-range pitches are reported as errors so obvious typos don't silently produce unplayable challenges.

### Example paste

```
title: Happy Birthday
group: melodies
description: slow, expressive
notes: C4 C4 D4 C4 F4 E4 - C4 C4 D4 C4 G4 F4

title: G major scale (trumpet-written)
group: scales
notes: G4 A4 B4 C5 D5 E5 F#5 G5

title: G major scale (concert)
group: scales
transpose: 0
notes: G4 A4 B4 C5 D5 E5 F#5 G5
```

## LLM prologue

Shipped as a constant, rendered read-only in the dialog, with a "Copy prologue" button.

```
You are helping generate exercises for a trumpet training app.
Output ONLY text matching the grammar below — no prose, no code fences.

GRAMMAR
  challenge := title_line group_line [description_line] [transpose_line] notes_line
  Multiple challenges: concatenate, separated by a blank line or just start
  the next challenge with a new `title:` line.

  title_line       = "title: "        <1-60 printable chars>
  group_line       = "group: "        ("long-tones" | "scales" | "melodies")
  description_line = "description: "  <up to 120 chars>
  transpose_line   = "transpose: "    <integer, e.g. -2 or 0>
  notes_line       = "notes: "        <NOTE> (" " <NOTE>)*
  NOTE             = PITCH | "-"
  PITCH            = [A-G] ("b" | "#")? [0-8]       (scientific pitch; C4 = middle C)
  "-"              = a rest

CONVENTIONS
  - Notes are written as they'd appear on a Bb trumpet part (WRITTEN pitch).
  - The app converts to concert pitch using `transpose`. Default is -2 (Bb trumpet).
  - If the user explicitly asks for CONCERT pitch, add `transpose: 0`.
  - Keep each exercise musical and short (4-32 notes is typical).
  - Use rests (`-`) between repeated pitches in melodies so the app can detect
    re-attacks (e.g. "E E E" should be "E4 - E4 - E4").

SELF-CHECK before answering, verify every line matches one of these regexes:
  ^title:\s+\S.{0,59}$
  ^group:\s+(long-tones|scales|melodies)$
  ^description:\s+\S.{0,119}$
  ^transpose:\s+-?\d+$
  ^notes:\s+([A-G][b#]?[0-8]|-)(\s+([A-G][b#]?[0-8]|-))*$

EXAMPLE
  title: Happy Birthday
  group: melodies
  description: slow, expressive
  notes: C4 C4 D4 C4 F4 E4 - C4 C4 D4 C4 G4 F4

Now generate: <USER FILLS IN HERE>
```

## Architecture

### New modules

**`src/training/custom-challenge-dsl.ts`** — pure parser/validator. No DOM, no storage.

```ts
export interface ParseError { line: number; message: string; }
export interface ParsedChallenge {
  challenge: Challenge;  // fully resolved concert-MIDI targets
  source: string;        // original DSL text for this block, for round-trip editing
}
export interface ParseResult {
  challenges: ParsedChallenge[];
  errors: ParseError[];
}
export function parseChallenges(source: string): ParseResult;
export function pitchToMidi(token: string): number | null;  // written MIDI; null if invalid token
```

Behaviour:
- Tokenize by line (keep line numbers 1-indexed for error reporting).
- State machine: on `title:` → flush in-progress record (validate complete: must have `group` and `notes`); start a new one.
- Unknown keys → error on that line.
- Duplicate keys within a block → error.
- Missing required keys at flush → error pointing to the block's `title:` line.
- Out-of-range concert MIDI → error on the `notes:` line.
- Parser returns **all** errors in one pass (not first-only) so the user can fix everything before re-pasting.

**`src/training/custom-storage.ts`** — localStorage shim for persistence.

```ts
export interface StoredCustomChallenge {
  id: string;
  title: string;
  group: ChallengeGroup;
  description?: string;
  targets: Target[];       // resolved concert MIDI
  source: string;          // raw DSL text, for edit round-trip
  createdAt: number;
}
export function listCustom(): StoredCustomChallenge[];   // newest first
export function addCustom(c: StoredCustomChallenge): void;
export function deleteCustom(id: string): void;
```

- localStorage key: `trumpet-custom-challenges-v1`.
- Same resilience pattern as existing `storage.ts` (try/catch on read and write; invalid JSON falls back to empty).
- No `updateCustom` — edit is `delete + add` at the UI layer (one code path, auto-resets stored best score via new id).

**`src/components/CustomChallengeDialog.tsx`** — add/edit dialog. Props:

```ts
interface Props {
  mode: "add" | "edit";
  initialSource?: string;    // prefilled in edit mode
  onClose: () => void;
  onSave: (added: StoredCustomChallenge[]) => void;
}
```

Dialog layout (stacked, modeled after `SettingsDialog`):

1. **Prologue section.** Read-only `<textarea>` showing the prologue constant; a `Copy prologue` button. Collapsible header so it doesn't dominate the dialog after first use (default: expanded on add, collapsed on edit).
2. **Paste area.** Editable `<textarea>` with placeholder "Paste the challenge(s) your LLM generated". Prefilled from `initialSource` in edit mode.
3. **Preview / errors panel.** Re-renders on every keystroke (debounced ~150ms):
   - Clean parse → list of blocks to be added: "Will add: Happy Birthday (melodies, 12 notes)". **Save**/**Add** button enabled.
   - Errors → list of `Line N: message` rows. Save/Add disabled.
4. **Footer buttons.** Cancel + Add (or Save in edit mode).

**Edit mode constraints:** Paste area must parse to exactly one challenge. Multiple blocks → error "Edit mode accepts a single challenge — remove extra blocks, or cancel and use Add". Save performs `deleteCustom(oldId)` then `addCustom(newChallenge)`.

### Modified modules

**`src/training/challenges.ts`** — extend the `ChallengeGroup` union with `"custom"` (only used as a group label; custom challenges carry their own group like `melodies`, so `"custom"` is only needed if we want a separate display group — see Picker section below).

> **Decision:** Custom challenges go into their own "Custom" display group; `group:` within the DSL is used to sort inside that group (long-tones first, then scales, then melodies). So the `ChallengeGroup` union does not need a new member for storage; only the `TrainingScreen`'s picker-group enum needs an extra `"custom"` bucket. Keeping `ChallengeGroup` as-is in `challenges.ts` keeps built-ins clean.

**`src/components/TrainingScreen.tsx`**

- Track custom challenges in a signal: `const [customs, setCustoms] = createSignal(listCustom())`.
- Extend `GROUP_ORDER` with a new picker-level bucket `"custom"` (distinct from the `ChallengeGroup` union used by challenges themselves).
- Extend `groupedChallenges` memo: built-ins go into their declared groups; customs go into the `"custom"` bucket, sorted by `createdAt` descending (newest first), with a secondary sort by their original `group` for grouping.
- Render a `+ Add` card at the start of the Custom group. Click → opens `CustomChallengeDialog` in `add` mode.
- On each custom card, render pencil and × buttons (small, top-right):
  - Pencil → opens dialog in `edit` mode with `initialSource` from storage.
  - × → two-stage confirm: first tap flips label to "Remove?" for 2s; second tap within the window calls `deleteCustom` and refreshes signal.
- After any add/edit/delete, refresh `setCustoms(listCustom())`.

**`src/components/ChallengeCard.tsx`** — add optional `onEdit` and `onDelete` props. When present, render the controls; when absent (built-ins), card is unchanged.

**`src/index.css`** — styles for:
- Add/Edit dialog (follows existing dialog patterns).
- Custom card edit/delete controls (small icons, top-right corner, subtle until hover).
- "Remove?" confirmation state for delete button.

## ID generation

`custom-<slug(title)>-<4-hex-random>`

- `slug(title)` = lowercase, non-alphanumeric → `-`, collapse repeats, trim leading/trailing `-`, max 32 chars.
- Random suffix avoids collisions between user challenges that share a title.
- `custom-` prefix reserves a namespace distinct from built-in ids.

## Validation error catalogue

| Case | Line reported | Message |
|---|---|---|
| Line doesn't match any known key | that line | `Unexpected line. Expected one of: title:, group:, description:, transpose:, notes:` |
| Duplicate key within a block | second occurrence | `Duplicate '<key>:' within the same challenge` |
| Missing required key | block's `title:` line | `Missing required field '<key>'` |
| Invalid group value | that `group:` line | `Unknown group '<val>'. Expected long-tones, scales, or melodies.` |
| Invalid pitch token | that `notes:` line | `Invalid note '<token>'. Expected e.g. C4, Bb3, F#5, or -.` |
| Out-of-range pitch | that `notes:` line | `Note '<token>' resolves to concert MIDI <n>, outside supported range 36-96.` |
| No note tokens | that `notes:` line | `At least one note is required.` |
| Transpose not an integer | that `transpose:` line | `Transpose must be an integer (e.g. -2, 0, 5).` |
| Title too long / empty | that `title:` line | `Title must be 1-60 characters.` |
| Description too long | that `description:` line | `Description must be 120 characters or fewer.` |
| Orphaned fields before any `title:` | first orphaned line | `Fields must follow a 'title:' line.` |

## Data flow — add

1. User clicks **+ Add** → dialog opens in `add` mode.
2. User clicks **Copy prologue** → clipboard receives prologue constant.
3. User goes to their LLM, pastes prologue, appends a request, gets output back.
4. User pastes output into the paste area; preview panel parses on each change.
5. On clean parse, user clicks **Add** → for each parsed block, generate id, build `StoredCustomChallenge`, call `addCustom`.
6. Dialog closes; picker refreshes; new cards appear at the top of the Custom group.

## Data flow — edit

1. User clicks pencil on a custom card → dialog opens in `edit` mode with `initialSource` prefilled.
2. User modifies the DSL in the paste area; preview panel re-parses.
3. On clean parse (and exactly one block), user clicks **Save** → `deleteCustom(oldId)` then `addCustom(newChallenge)`. Best score for old id is orphaned (effectively reset — this is the desired behavior for an edited exercise).
4. Dialog closes; picker refreshes; the edited challenge moves to the top (newest first sort).

## Testing

**`custom-challenge-dsl.test.ts`**

- Single valid block → one resolved challenge, correct targets.
- Multi-block paste, with and without blank-line separators.
- All field edge cases: accidentals (`b`, `#`), rest token `-`, transpose `0`, negative transpose, no-description block, no-transpose block (default applied).
- Regex boundaries: octave 8 accepted, octave 9 rejected, lowercase `c4` rejected, `Bb10` rejected, `H5` rejected, `C` (no octave) rejected.
- Every error case from the catalogue reports the right line number and message.
- Multiple errors across a single paste are all reported in one pass.
- Range clamp error fires on obviously-too-low/too-high pitches.

**`custom-storage.test.ts`**

- `listCustom()` empty when nothing stored; after `addCustom` returns the record; after `deleteCustom` the record is gone.
- Round-trip: write N, read N in newest-first order.
- Corrupt JSON in localStorage → `listCustom()` returns `[]`, no throw.
- Write failure (stubbed) → swallowed silently, matching existing `storage.ts` pattern.

**Integration (light)**

- Seed localStorage with a custom challenge, mount `TrainingScreen`, assert the Custom group renders it.
- Click `+ Add`, paste a valid block, click Add, assert the challenge appears and localStorage updated.

**Existing tests** (`challenges.test.ts`, `training-engine.test.ts`, etc.) stay untouched and must continue to pass.

## Non-goals

- Multi-device sync / sharing / export.
- Editing by splitting an existing multi-block paste.
- Rich validation like "trumpet range" (we only clamp to broad concert-MIDI range).
- Preventing title collisions across custom challenges (the random id suffix handles collision; multiple challenges titled "Warmup" are allowed).
- A "reset best score" UI — edit already clears it implicitly.
